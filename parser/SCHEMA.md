# Berkeley Swim Schedule JSON Schema

This document describes the JSON files generated by the Java parser and consumed by the web app:

- `web/public/data/king.json`
- `web/public/data/west.json`

The parser writes one file per pool. Each file contains pool metadata, season metadata, known closures, and a flattened list of schedule entries.

## Shape

```json
{
  "pool": {
    "id": "king",
    "name": "King Pool",
    "address": "1700 Hopkins St., Berkeley, CA 94707",
    "phone": "510-981-5105"
  },
  "season": {
    "label": "Spring 2026",
    "valid_from": "2026-03-02",
    "valid_through": "2026-06-07"
  },
  "last_updated": "2026-02-24T00:00:00Z",
  "closures": [
    {
      "date": "2026-03-30",
      "reason": "City holiday"
    }
  ],
  "schedule": [
    {
      "day": "Monday",
      "activity": "Lap Swim",
      "start": "07:30",
      "end": "12:30",
      "notes": ""
    }
  ]
}
```

## Fields

### `pool`

Metadata for the pool represented by this file.

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | string | yes | Stable pool identifier. Current values: `king`, `west`. |
| `name` | string | yes | Human-readable pool name. |
| `address` | string | yes | Full street address for display. |
| `phone` | string | yes | Pool phone number in `510-000-0000` style. |

Coordinates are intentionally deferred. When added later, use optional numeric fields `pool.lat` and `pool.lng`.

### `season`

Metadata for the schedule season covered by the source PDF.

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `label` | string | yes | Display label, such as `Spring 2026`. |
| `valid_from` | string | yes | First schedule date in `YYYY-MM-DD` format. |
| `valid_through` | string | yes | Last schedule date in `YYYY-MM-DD` format. |

### `last_updated`

ISO-8601 timestamp derived from the PDF's "Last Revised" date.

Example: `2026-02-24T00:00:00Z`

### `closures`

Array of dates when the pool should be treated as closed even if schedule entries exist.

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `date` | string | yes | Closure date in `YYYY-MM-DD` format. |
| `reason` | string | yes | Short display reason, such as `City holiday` or `Pre-Summer Staff Training`. |

### `schedule`

Flattened schedule entries. Each time range becomes one object, so one activity can have multiple entries on the same day.

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `day` | string | yes | Full English day name: `Monday` through `Sunday`. |
| `activity` | string | yes | Activity label from the PDF, such as `Lap Swim` or `Family Swim`. |
| `start` | string | yes | Start time in 24-hour `HH:mm` format. |
| `end` | string | yes | End time in 24-hour `HH:mm` format. |
| `notes` | string | yes | Empty string when there is no note. Currently `Limited Lap Lanes` for `**` markers. |

## Consumer Rules

- Treat `closures` as overriding normal schedule entries for the same date.
- Do not infer a date from a `schedule` entry by itself; combine `day` with the active season/date context in the web app.
- Do not rely on array order for business logic. Sort by day and start time in the consumer when needed.
- Preserve unknown additive fields when transforming data, so future optional fields like `pool.lat` and `pool.lng` do not break consumers.
